## Repository 1

**Repository Name:** PROJ4JS  
**Link:** https://github.com/proj4js/proj4js

### Purpose
What does this project do?
PROJ4JS is a JavaScript library that transforms point coordinates from one coordinate system to another.

### Repository Structure
What folders or files stand out?
This project has a lot of the same folders and files that were introduced to us in lecture, including a README, a license, and a .gitignore file. I also noticed that it has documents for the authors with contact information, a reference document, and even a security policy.

### Documentation
How helpful is the README?
The README clearly explains what the project does, but more than that it has instructions and examples of how to install and run the code. There are samples of code showing how to do basic projections, named projections, datum transformations, etc.

### Activity
Is the repository actively maintained?
Yes, the repository is actively maintained, some of the files have been updated as recently as 5 days ago. 

### Key Observation
One thing this project does well.
This project does well keeping things simple and clear. The README gets straight to the point with an explanation and lots of examples. This seems to be a relatively simple tool, it has one job, so the minimal repository set up makes sense.

## Repository 3

**Repository Name:** osmnx
**Link:** https://github.com/gboeing/osmnx

### Purpose
What does this project do? OSMnx is a Python package to download, model, analyze, and visualize street networks and other geospatial features from OpenStreetMap.

### Repository Structure
What folders or files stand out? The repository has the basic files and folders that I've seen in previous repositories. This includes a README, a license, and citation files as well as folders like docs. Interestingly, the README links a getting started guide, tutorials to work through, and an installation guide that are all housed outside of github. This is different from the last two repositories I looked at, where that information was stored directly in github.

### Documentation
How helpful is the README? The README is simple, but very helpful because it contains the links to the outside documentation like the getting started and installation guides. 

### Activity
Is the repository actively maintained? I would say this repository is actively maintained, the last update was just a couple of weeks ago. 

### Key Observation
One thing this project does well. The OSMnx repository does a good job of keeping things streamlined on github and all about the tool itself. The more complicated documentation is housed somewhere else, and 'how-to' or usage questions are directed outside of github as well.

## Reflection

For the most part it seems open source projects are organized in similar ways. They contain a lot of the same files that are standard across projects, like a README or .gitignore file. How additional information is stored and presented changes project to project. Some repositories store everything in folders, some use the github wiki, and some take advantage of non-github platforms. It doesn't seem like there is a right or wrong way, instead I think the organization of an open source project is based on how many things the tool acomplishes. A simple project wouldn't need to make use of lots of files and folders or a separate place to store information, while a more complex one might benefit from that.
